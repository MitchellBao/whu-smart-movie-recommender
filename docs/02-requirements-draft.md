# 第二部分 项目需求规格说明书

## 2.1 用户需求说明

本系统以电影推荐为实验场景，面向普通用户在海量电影信息中发现感兴趣内容的需求，提供个性化电影推荐、评分反馈、推荐理由解释和自然语言查询能力。

系统核心需求集中在三个方面：

1. 个性化推荐生成：基于用户评分和电影元数据生成 Top-N 推荐电影列表。
2. 智能语义解释：通过大语言模型或默认解释机制，为推荐结果生成自然语言理由。
3. 交互与反馈闭环：用户可以提交评分，系统根据评分数据刷新推荐结果，并将推荐结果持久化保存。

## 2.2 用户角色

| 角色 | 角色说明 | 主要需求 |
|---|---|---|
| 普通用户 | 使用系统获取电影推荐的用户 | 浏览电影、提交评分、查看推荐列表、查看推荐理由、进行自然语言查询 |
| 前端调用方 | 前端页面或测试端 | 调用后端 API，展示推荐结果、评分状态和问答结果 |
| 系统管理员 | 系统维护者 | 管理用户、电影、评分和推荐数据，维护服务运行状态 |
| 开发维护人员 | 项目开发成员 | 维护前端、后端、算法服务、数据库、部署脚本和接口文档 |

## 2.3 业务需求概述

### 2.3.1 个性化推荐

系统应根据用户评分数据生成个性化推荐结果。推荐结果应体现用户兴趣偏好，并以结构化 JSON 形式返回给后端或前端。

推荐结果至少包括：

- 推荐电影编号。
- 推荐电影标题。
- 推荐电影类型。
- 推荐预测得分。
- 推荐理由。

### 2.3.2 推荐解释

系统应支持对推荐结果进行解释。推荐解释可以来源于算法侧说明，也可以来源于大语言模型生成的自然语言文本。

推荐解释应满足：

- 使用中文自然语言表达。
- 内容简洁，便于用户理解。
- 能够持久化保存到数据库。
- 当 LLM 未启用或调用失败时，仍能返回默认解释。

### 2.3.3 评分反馈

系统应支持用户提交电影评分。评分数据应保存到数据库，并作为推荐系统的重要用户行为数据。

评分提交后，系统应能够触发推荐结果刷新，使评分反馈进入推荐闭环。

### 2.3.4 自然语言查询

系统应支持简单自然语言查询。例如：

```text
推荐一部烧脑科幻片
我想看轻松一点的喜剧
有没有适合周末看的电影
```

后端应结合当前推荐结果和候选电影，返回简短回答。

## 2.4 功能需求

### 2.4.1 用户信息管理

系统应保存用户基础信息，用于区分不同用户的评分行为和推荐结果。

字段需求：

| 字段 | 说明 |
|---|---|
| user_id | 用户唯一编号 |
| username | 用户名 |
| password | 密码字段 |
| gender | 性别 |
| age | 年龄 |

### 2.4.2 电影信息管理

系统应保存电影基础信息，用于推荐展示和推荐解释。

字段需求：

| 字段 | 说明 |
|---|---|
| movie_id | 电影唯一编号 |
| title | 电影标题 |
| release_year | 上映年份 |
| genres | 电影类型 |
| director | 导演 |

### 2.4.3 用户评分管理

系统应支持用户对电影提交评分。

字段需求：

| 字段 | 说明 |
|---|---|
| rating_id | 评分记录编号 |
| user_id | 用户编号 |
| movie_id | 电影编号 |
| score | 评分值 |
| timestamp | 评分时间戳 |

### 2.4.4 推荐结果管理

系统应保存推荐算法生成的推荐结果。

字段需求：

| 字段 | 说明 |
|---|---|
| id | 推荐记录编号 |
| user_id | 用户编号 |
| movie_id | 推荐电影编号 |
| predicted_score | 算法预测得分 |
| reason | 推荐理由 |

其中 `reason` 字段用于保存大语言模型生成或算法返回的推荐解释，是系统支持可解释推荐的关键字段。

### 2.4.5 推荐算法服务

系统应提供独立算法服务，完成推荐计算。

算法服务需求包括：

- 支持健康检查。
- 支持根据 `userId` 和 `topN` 返回推荐结果。
- 支持协同过滤思想。
- 支持 SVD 或矩阵分解排序思想。
- 返回结构化 JSON 数据。

### 2.4.6 后端业务服务

后端服务应负责业务流程编排。

后端需求包括：

- 接收前端请求。
- 访问 MySQL 数据库。
- 调用 Python 算法服务。
- 调用 LLM 客户端。
- 聚合电影元数据、推荐得分和推荐理由。
- 向前端返回统一 JSON 响应。

### 2.4.7 前端展示

前端应负责系统结果展示和用户交互。

前端需求包括：

- 展示推荐电影列表。
- 展示电影标题、类型、推荐得分和推荐理由。
- 提供评分入口。
- 提供自然语言查询入口。
- 对接口异常和 LLM 降级文案进行友好提示。

## 2.5 接口需求

### 2.5.1 前后端交互接口

| 接口名称 | 请求路径 | 方法 | 参数 | 响应 | 说明 |
|---|---|---|---|---|---|
| 获取推荐列表 | `/api/recommend/movie` | GET | `userId`, `topN` | `{code, data}` | 获取指定用户推荐电影 |
| 提交评分 | `/api/rating/submit` | POST | `{userId, movieId, score}` | `{code, message}` | 用户提交电影评分 |
| 自然语言查询 | `/api/llm/query` | POST | `{userId, queryText}` | `{code, responseText, relatedMovies}` | 获取语义化推荐回答 |

### 2.5.2 后端与算法服务接口

| 接口名称 | 请求路径 | 方法 | 参数 | 响应 | 说明 |
|---|---|---|---|---|---|
| 算法服务健康检查 | `/api/python/health` | GET | 无 | `{status}` | 检查算法服务状态 |
| 推荐计算 | `/api/python/calculate` | POST | `{userId, topN}` | `{userId, recommendations}` | 计算推荐结果 |

### 2.5.3 外部 LLM 接口

后端通过统一 LLM 客户端调用外部大语言模型服务。LLM 接口采用 OpenAI-compatible API 风格，可适配 OpenAI、DeepSeek、通义千问等服务。

Prompt 信息可包括：

- 用户信息摘要。
- 用户历史偏好。
- 推荐电影信息。
- 电影类型。
- 算法预测得分。

## 2.6 数据需求

### 2.6.1 数据来源

系统以 MovieLens 等公开数据集作为实验数据来源。MovieLens 数据集包含规范的用户评分记录和电影信息，适合用于推荐算法验证。

### 2.6.2 数据存储

系统采用 MySQL 存储核心业务数据。

核心数据表包括：

| 表名 | 说明 |
|---|---|
| users | 用户信息表 |
| movies | 电影信息表 |
| ratings | 用户评分表 |
| recommendations | 推荐结果表 |

### 2.6.3 数据链路

系统数据链路如下：

```text
用户评分数据
    ↓
推荐算法计算
    ↓
推荐结果生成
    ↓
LLM 推荐理由生成
    ↓
recommendations 表持久化
    ↓
前端展示
```

## 2.7 运行环境需求

| 模块 | 环境需求 |
|---|---|
| 前端 | Node.js、Vue.js、Vite |
| 后端 | JDK 17、Maven、Spring Boot |
| 算法服务 | Python、FastAPI、uvicorn、pandas、numpy、scikit-learn |

## 2.8 非功能需求

### 2.8.1 可维护性

系统应采用清晰分层结构，降低模块耦合。后端应划分 controller、service、repository、entity、dto 等模块。

### 2.8.2 可扩展性

系统应支持算法服务独立迭代，LLM 服务商可通过统一客户端切换。整体架构应支持替换数据源和扩展推荐场景。

### 2.8.3 可解释性

系统应保存推荐理由，使推荐结果具有可解释性。推荐解释不应破坏推荐算法主流程。

### 2.8.4 可运行性

系统应提供本地启动、停止和测试脚本，支持快速验证后端、算法服务、数据库和接口主链路。

### 2.8.5 容错性

当 LLM 未配置或调用失败时，系统应返回默认解释文本，保证推荐接口仍可正常使用。

### 2.8.6 协作可追踪性

项目代码和文档应通过 Git 管理，提交记录应能够反映成员贡献，便于课程过程考核。
