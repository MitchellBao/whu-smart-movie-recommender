# 第一部分 选题及团队组建报告

## 1.1 选题目的和意义
随着数字内容规模的持续增长，用户在海量信息中高效发现符合自身兴趣的内容已成为信息系统设计中的重要问题。推荐系统通过对用户历史行为数据的分析与建模，实现个性化的信息过滤与内容分发，已成为视频平台、电商平台以及内容社区等互联网服务中的关键技术。本项目旨在设计并实现一个面向用户行为数据的个性化推荐系统框架，通过算法建模与系统工程实践，探索从用户行为数据到推荐结果生成的完整技术流程。

本项目不仅关注推荐结果本身的生成，更重视推荐系统内部的数据处理与算法实现机制。系统将以公开数据集为基础，构建从数据清洗、特征构建到模型训练与推荐生成的完整数据处理流程。在算法层面，团队将实现协同过滤与矩阵分解等经典推荐算法，通过代码级实现与实验评估，深入理解推荐模型在用户兴趣预测中的作用机制。

在系统设计方面，本项目尝试将传统推荐算法与大语言模型能力结合。推荐算法负责从用户行为数据中计算潜在兴趣关联，而大语言模型则用于对推荐结果进行语义化解释，并支持基于自然语言的交互式查询，从而在保证推荐准确性的同时提升系统的可解释性与交互体验。这种“计算结果 + 语义表达”的结合，使系统不仅能够给出推荐列表，也能够对推荐逻辑进行直观说明。

在应用层面，系统以电影推荐作为实验场景进行实现与验证，但整体架构采用通用的 User–Item 推荐模型设计。通过对数据层与推荐引擎的模块化设计，在替换数据源与特征配置后，系统能够扩展至图书推荐、音乐推荐或电商商品推荐等其他应用领域。

通过本项目的实践，团队将综合运用数据处理、算法实现与软件工程方法，完成一个从数据输入到推荐输出的完整系统原型，并在工程实现过程中提升对推荐系统底层机制与系统架构设计的理解。

## 1.2 研究内容、研究方法、技术路线及可行性分析
### 1.2.1 研究内容
本项目旨在构建一个完整的个性化推荐系统原型，并以电影推荐作为实验场景进行系统实现与验证。整体研究内容主要包括数据处理、推荐算法实现以及智能交互三个方面。

在数据工程层面，系统以 MovieLens 等公开数据集为基础，构建用户行为数据的处理流程，包括数据清洗、评分数据整理以及用户与物品特征的结构化表示。通过对用户历史评分记录和物品属性信息进行组织与处理，形成基础的用户画像与物品特征，为后续推荐模型提供稳定的数据输入。同时，在数据处理过程中关注推荐系统常见的数据稀疏问题，并通过合理的数据组织方式提高数据利用效率。

项目将实现经典的协同过滤方法，包括基于用户的协同过滤（User-Based Collaborative Filtering）和基于物品的协同过滤（Item-Based Collaborative Filtering）。在此基础上，进一步实现基于矩阵分解的推荐模型，通过潜在因子分解方法刻画用户兴趣与物品特征之间的隐含关系，并利用 RMSE 等指标对模型预测效果进行评估，从而比较不同算法在推荐准确性上的表现。

项目尝试引入大语言模型能力，对推荐算法生成的结果进行语义增强。系统将通过提示词设计，将用户兴趣特征与推荐结果转化为自然语言形式的推荐说明，并支持简单的自然语言查询，从而提升推荐系统的可解释性与交互体验。

通过上述研究与实现，系统将形成一个从数据处理、推荐生成到推荐解释的完整技术流程，并在系统架构上保持良好的模块化设计，使其能够在替换数据源后扩展至图书推荐、电商商品推荐等其他应用场景。

### 1.2.2 研究方法
本项目遵循“理论分析、实验验证与工程实现相结合”的研究路径。首先，通过文献研究方法梳理推荐系统领域的经典算法原理以及相关技术的发展现状，为系统设计提供理论基础。在此基础上，确定系统的总体技术框架与核心算法方案。

首先，我们采用实验分析方法对推荐模型进行训练与评估。通过划分训练集与测试集，对协同过滤与矩阵分解等算法进行实现与调优，并在不同参数设置下比较模型的预测性能，利用 RMSE 等评价指标对推荐效果进行量化分析，从而验证算法模型的有效性。

其次，项目采用软件工程方法进行开发管理。开发流程参考敏捷开发思想，将系统功能划分为多个迭代阶段逐步完成。在系统架构上采用前后端分离的设计模式：后端基于 Spring Boot 构建服务接口，负责推荐算法调用、数据处理及业务逻辑实现；前端基于 Vue 框架实现用户交互界面与数据可视化，通过 RESTful API 完成数据交互，从而保证系统结构的清晰性与可维护性。

### 1.2.3 技术路线
系统整体架构遵循高内聚、低耦合的设计原则，自底向上划分为数据层、算法层、服务层和应用层，各模块之间通过标准接口进行数据交互。

在数据层，系统以 MovieLens 公开数据集作为实验数据来源，利用 Pandas 与 NumPy 等 Python 数据处理库完成数据清洗、结构化整理与基础特征构建，为推荐算法提供稳定的数据输入。

在算法层，采用 Python 实现推荐系统核心逻辑，包括基于用户的协同过滤、基于物品的协同过滤以及矩阵分解模型，并将推荐算法封装为独立模块，以便在系统中进行调用与扩展。

在服务层，系统基于 Spring Boot 框架构建后端服务，负责用户数据管理、推荐请求处理以及推荐结果返回等核心业务逻辑。同时通过 API 接入大语言模型能力，将推荐结果与用户特征信息转换为提示输入，从而生成自然语言形式的推荐解释。

在应用层，前端采用 Vue.js 框架开发用户交互界面，通过 ECharts 组件实现用户兴趣与推荐结果的可视化展示，并提供基础的自然语言交互窗口以支持简单的推荐查询。

在系统部署与协作方面，项目采用 Docker 进行容器化部署，并通过 Git 进行代码版本管理与团队协作，从而保证开发流程的规范性与系统部署的可复现性。

### 1.2.4 可行性分析
本项目在数据来源、技术实现以及团队协作等方面均具备良好的可行性。

系统采用 MovieLens 公开数据集作为实验数据来源。该数据集在推荐系统研究中被广泛使用，数据结构规范、包含完整的用户评分记录和电影信息，能够满足推荐算法建模与实验验证的需要，同时避免了实际数据采集过程中可能涉及的成本与合规问题。

对于技术实现部分，推荐系统相关算法已经较为成熟，协同过滤与矩阵分解等方法具有清晰的实现路径。Python 生态提供了完善的数据处理与科学计算工具，如 Pandas 与 NumPy，可有效支持数据处理与算法实现。同时，大语言模型能力可以通过 API 方式进行调用，无需进行复杂的模型训练，从而降低系统实现的技术门槛。

在团队与开发资源方面，项目成员的分工覆盖算法实现、后端开发以及前端交互等主要开发环节，能够支撑系统的整体开发工作。课程安排提供约十四周的开发周期，时间上能够满足系统设计、实现与测试的基本需求，从而保证项目按计划完成。

## 1.3 项目特色与创新点
本项目在系统设计与工程实现方面具有一定的特色，主要体现在推荐算法与语言模型结合、核心算法自主实现以及系统架构的模块化设计等方面。

首先，在推荐结果呈现方式上，系统尝试将传统推荐算法与大语言模型能力进行结合。推荐算法负责从用户行为数据中计算潜在兴趣关联并生成推荐结果，而大语言模型则用于对推荐结果进行自然语言解释，使系统不仅能够给出推荐列表，还能够对推荐原因进行说明。这种方式在保证推荐准确性的同时提升了系统的可解释性与用户交互体验。

其次，在工程实现层面，项目尽量避免对现成推荐服务的直接调用，而是围绕推荐系统的核心流程进行实现。系统从数据处理开始构建完整的数据处理流程，包括数据清洗、特征整理以及推荐模型实现，并通过代码实现协同过滤与矩阵分解等核心算法。通过这种方式，团队能够更深入地理解推荐系统中数据处理、模型训练与推荐生成之间的关系。

此外，在系统架构设计方面，项目采用模块化设计思想，将数据处理模块、推荐算法模块、后端服务模块以及前端应用模块进行清晰划分，各模块之间通过标准接口进行交互。虽然系统以电影推荐作为实验场景进行实现，但整体架构保持了良好的扩展性，通过替换数据源与特征配置，系统能够扩展到图书推荐、电商商品推荐等其他应用场景。

通过上述设计，项目在保证系统功能完整性的同时，也兼顾了推荐系统的可解释性、工程实践深度以及系统架构的可扩展性。

# 第二部分 项目需求规格说明书

## 2.1 用户需求说明
本部分旨在详细阐述“面向大规模用户行为数据的智能电影推荐系统”的各类业务需求。系统以电影推荐为实验场景，旨在解决用户在海量数据中信息获取效率低的问题，同时通过大语言模型技术提升推荐结果的可解释性。

### 2.1.1 业务需求概述
本系统的核心业务需求主要集中在以下三个维度：
1. 个性化推荐生成：基于MovieLens数据集，通过协同过滤与矩阵分解算法，为用户生成符合其潜在兴趣的电影推荐列表。
2. 智能语义解释与交互：利用大语言模型（LLM）将推荐结果转化为自然语言解释，并支持用户通过自然语言查询进行交互，解决传统推荐系统“黑盒”问题。
3. 数据可视化与管理：提供用户画像、电影详情展示及推荐结果的可视化大屏，提升用户体验。

### 2.1.2 用户角色定义
系统主要涉及两类用户角色：

| 角色名称 | 描述 | 主要职责 |
| :--- | :--- | :--- |
| 普通用户 | 系统的主要服务对象，即电影浏览者 | 注册/登录系统、浏览电影信息、对电影进行评分、查看个性化推荐、查询推荐理由、与智能助手交互 |
| 系统管理员 | 系统的管理与维护者 | 管理用户账户信息、维护电影基础数据库、监控系统运行状态、管理算法模型参数配置 |

### 2.1.3 用户故事
遵循敏捷开发原则，系统主要功能通过以下用户故事进行描述：
1. **用户注册与画像构建**：作为一名未注册的访客，我想要注册账户并填写基本的兴趣偏好，以便于系统能够初始化我的用户画像，从而提供初步的个性化服务。
2. **电影浏览与评分**：作为一名登录用户，我想要浏览电影列表、查看电影详情（如导演、演员、类型）并对看过的电影进行评分，以便于丰富我的历史行为数据，使系统更精准地捕捉我的兴趣偏好。
3. **个性化推荐获取**：作为一名登录用户，我想要在首页查看到系统基于我的历史行为生成的个性化电影推荐列表，以便于高效地发现我可能感兴趣但未曾关注的电影，降低信息检索成本。
4. **推荐理由的语义化解释**：作为一名登录用户，我想要点击推荐电影旁边的“为什么推荐？”按钮，获得一段自然语言的解释（例如：“推荐这部电影是因为您之前对科幻类电影评分较高，且该导演的作品风格与您的观影历史高度匹配”），以便于我理解系统的推荐逻辑，增加对推荐结果的信任度。
5. **智能自然语言交互**：作为一名登录用户，我想要在搜索框中输入自然语言描述（例如：“我想看一部类似于《盗梦空间》烧脑的电影”），以便于跳过复杂的筛选步骤，直接获得符合特定语义语境的推荐结果。
6. **数据可视化展示**：作为一名登录用户，我想要查看个人兴趣分布的雷达图或评分历史统计图，以便直观地了解自己的观影偏好分布情况。

### 2.1.4 核心业务流程分析
本系统核心业务流程涵盖从用户行为输入到智能推荐输出的完整闭环，具体流程如下：

**流程一：推荐生成与解释流程**
1. 行为采集：用户在前端界面进行浏览、评分、收藏等操作，行为数据通过API传输至后端服务层。
2. 数据处理：ETL模块定时对行为数据进行清洗与特征构建，更新用户-物品评分矩阵。
3. 算法计算：算法层调用协同过滤或矩阵分解模型，计算用户潜在兴趣，生成候选推荐列表。
4. 智能增强：服务层将推荐结果及用户特征转化为Prompt，调用大语言模型API，生成推荐理由的自然语言描述。
5. 结果展示：前端接收包含电影详情与推荐理由的数据，通过可视化组件渲染展示给用户。

**流程二：自然语言交互查询流程**
1. 输入：用户在交互窗口输入自然语言查询语句。
2. 意图识别：系统通过大语言模型解析用户意图（如：查询特定类型电影、查询演员作品、寻求特定场景推荐）。
3. 混合检索：系统结合传统数据库检索与向量相似度检索，筛选匹配电影。
4. 输出：系统将匹配结果返回前端，并附带基于语义理解的回复文本。

## 3.4 复杂功能的算法设计
为了解决单纯基于用户的协同过滤在数据稀疏场景下导致的“冷启动冷门推荐精度差”问题，本系统在核心微服务端设计并实现了一种混合型推荐矩阵算法。该算法融合了协同过滤的邻域关系优势与奇异值分解（SVD）的隐语义模型能力，以提升推荐系统的准确性与覆盖率。

### 3.4.1 算法设计思路
本算法采用两阶段策略：
1. **召回阶段**：利用皮尔逊相关系数计算目标用户与邻居用户的相似度，选取Top-K高相似度邻居，根据其偏好构建初始的候选电影池，解决“找谁来看”的问题。
2. **排序阶段**：引入隐语义模型，通过SVD矩阵分解提取用户与电影的潜在特征向量，对候选池中的电影进行精确评分预测，并结合全局偏置项进行最终排序，解决“怎么打分”的问题。

### 3.4.3 核心伪代码实现
以下伪代码展示了混合推荐算法的核心数据清洗与推演逻辑：
```
function generate_hybrid_recommendation(target_user_id, rating_matrix):
       # 步骤1: 获取高相似度邻居用户 (基于皮尔逊相关系数)
       # 计算目标用户与所有其他用户的相关系数
       neighbors = calculate_pearson_correlation(target_user_id, rating_matrix)
       # 截取相似度最高的前K个用户作为邻居圈层
        top_k_neighbors = extract_top_k(neighbors, k=20)

       # 步骤2: 根据邻居偏好建立候选电影池
       # 获取邻居看过但目标用户未看过的电影，作为待排序的候选集
        candidate_movies = get_unseen_movies(target_user_id, top_k_neighbors)

       # 步骤3: 引入隐语义模型处理极度稀疏矩阵的数据平滑
       # 对评分矩阵进行奇异值分解，降维提取潜在特征 (维度设为50)
       U, Sigma, V_T = perform_SVD_decomposition(rating_matrix, latent_features=50)

       # 步骤4: 结合奇异值计算用户对候选池影片的打分预测
       predicted_scores = []
        for movie in candidate_movies:
                # 计算全局偏置，如电影的热门程度或平均分基线
                base_bias = calculate_global_bias(movie)
                # 利用分解后的矩阵计算用户潜在兴趣与电影特征的匹配分
                svd_score = dot_product(U[target_user_id], V_T[movie.id])
                # 最终得分为基线偏置与隐因子匹配分之和
                final_score = base_bias + svd_score
                predicted_scores.append( (movie, final_score) )

       # 步骤5: 按预测得分降序排列并截取 Top-N
       # 返回得分最高的N部电影作为最终推荐结果
        return sort_and_slice(predicted_scores, N=10)
```

## 3.5 面向对象方法类图的详细设计
本系统后端基于 Spring Boot 框架，采用面向对象编程（OOP）思想进行业务层设计。为了保障系统的可维护性与扩展性，设计了针对推荐引擎业务流程的经典控制/实体防腐层聚合根结构。

### 3.5.1 类图设计解读
- **实体类**：User（用户）、Movie（电影）、Rating（评分）构成了系统的核心业务实体。它们是强校验的 POJO，负责与 ORM 映射，承载系统运行时的状态数据，并封装了基础的数据完整性校验逻辑。
- **控制行为类**：
  - RecommenderService：作为推荐引擎对外暴露的门面。它封装了复杂的业务编排逻辑，聚合了对 Python 端算法计算层的下发能力。
  - LLMClient：大语言模型调用抽象类。通过依赖注入机制实现松耦合，允许系统灵活切换不同的 LLM 服务商。
  - RecommenderController：作为入口控制器，负责接收前端 HTTP 请求并调用 Service 层进行处理。

通过这种设计，RecommenderService 在一套接口上下文中流水线式地组装起“底层算推荐 -> 高层大模型分析解读理由”的一体化服务，实现了计算逻辑与语义表达的解耦。

## 3.6 接口设计
本系统遵循前后端分离架构，后端基于 Spring Boot 提供 RESTful API，前端基于 Vue.js 调用接口。同时，系统内部涉及 Python 算法服务与 Java 后端的通信，以及大语言模型（LLM）API 的调用。

### 3.6.1 前后端交互接口 (RESTful API)

| 接口名称 | 请求路径 | 请求方法 | 请求参数 (JSON) | 响应数据 (JSON) | 说明 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 用户登录 | /api/user/login | POST | {username, password} | {code, token, message} | 验证用户身份，返回 JWT Token |
| 获取推荐列表 | /api/recommend/movie | GET | userId (Query) | {code, data: [{movieId, title, genres, score, reason}]} | 调用推荐引擎，获取包含大模型解释的推荐列表 |
| 提交评分 | /api/rating/submit | POST | {userId, movieId, score} | {code, message} | 用户对电影进行评分，触发数据更新 |

### 3.6.2 内部服务接口 (Java <-> Python)
由于核心推荐算法由Python实现，后端服务由 Java 实现，两者通过 HTTP 或文件系统进行交互。
**调用逻辑：**
1. Java (Spring Boot) 接收前端请求，预处理数据并写入数据库。
2. Java 调用 Python API (POST /api/python/calculate)，传递用户ID或触发信号。
3. Python 读取数据库中的评分表，执行 ETL 和算法计算。
4. Python 将生成的推荐结果（Top-N 列表）写回数据库或直接返回 JSON 给 Java。

### 3.6.3 外部服务接口 (LLM API)
**服务提供商**： OpenAI API / 通义千问 / 文心一言 (配置在 application.yml 中)
**交互方式**： HTTPS POST
**Prompt 设计示例**：

## 3.8 UI（界面）设计
系统前端基于 Vue.js 和 ECharts 构建，旨在提供直观的可视化体验和智能交互。

### 3.8.1 设计风格
- **风格**： 现代化暗色系（符合电影浏览场景），搭配亮色（如青色）作为高亮色。
- **布局**： 采用栅格化布局（Grid/Flexbox），顶部导航栏 + 侧边栏（可选）+ 主内容区。

### 3.8.2 核心页面原型描述
1. **首页 (Home/Dashboard)**
   - 顶部：搜索框（支持自然语言输入）、用户头像下拉菜单。
   - 左侧 (70%)：个性化推荐流，卡片式布局展示推荐电影。交互元素：每张卡片包含“为什么推荐？”按钮（点击弹出 Modal 显示 LLM 生成的解释）。
   - 右侧 (30%)：数据可视化，使用 ECharts 渲染的雷达图（用户兴趣分布：动作、爱情、科幻等占比）。热门榜单：实时热门电影 Top 10。
2. **电影详情页 (Movie Detail)**
   - 布局：左侧电影海报与简介，右侧评分与评论。
   - 功能：显眼的评分组件（1-5颗星），用户点击后即时提交评分并触发推荐更新。
3. **智能交互面板 (Smart Assistant)**
   - 位置：页面右下角悬浮的圆形按钮，点击展开。
   - 样式：类似 ChatGPT 的对话窗口。
   - 功能：用户输入框 + 发送按钮。历史对话记录区显示之前的问答（如“我想看烧脑的电影” -> 系统推荐《盗梦空间》并解释）。
4. **管理员后台 (Admin Panel)**
   - 布局：标准的 Admin LTE 风格。
   - 功能：数据表格展示用户列表、电影列表；按钮支持增删改查（CRUD）；系统运行状态监控图表（CPU/内存使用率模拟）。

### 3.8.3 关键组件交互说明
- **评分反馈机制**：用户评分后，前端通过 Axios 发送 POST 请求，后端接收后不仅更新数据库，还会向 Python 算法服务发送信号，触发增量更新或重新计算推荐列表，前端“推荐流”区域局部刷新。
- **大模型解释加载**：点击“为什么推荐？”时，前端显示“AI思考中...”的 Loading 动画，待 LLM 接口返回 JSON 后，动态渲染解释文本。
