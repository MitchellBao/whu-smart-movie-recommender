# 智能电影推荐系统答辩讲稿

各位老师好，我们小组的课程项目是“面向大规模用户行为数据的智能电影推荐系统”。我是鲍明頡，主要负责项目总体架构、后端主链路、数据库设计、前后端联调、DeepSeek 接入、数据可视化和最终功能整合。

## 1. 项目目标

本项目要解决的问题是：用户面对大量电影时，很难快速找到适合自己的内容。因此我们设计了一个智能推荐系统，基于 MovieLens 用户评分数据，为用户生成个性化电影推荐，并通过大语言模型生成更容易理解的推荐解释。

系统的核心思路可以概括为：

```text
用户评分数据 -> 推荐算法计算 -> 后端业务编排 -> 前端展示推荐 -> LLM 解释推荐原因
```

也就是说，算法负责“算得准”，大模型负责“讲得清楚”。

## 2. 系统架构

系统采用前后端分离和分层解耦架构，主要分为四层：

- 前端应用层：Vue 3 + Vite，负责用户登录、电影浏览、评分、推荐展示、画像展示和智能问答。
- 后端服务层：Spring Boot，负责用户、电影、评分、推荐、显式偏好、数据看板和 LLM 问答接口。
- 算法服务层：Python FastAPI，直接读取 MySQL 中的 MovieLens 评分数据，计算推荐结果。
- 数据层：MySQL，保存用户、电影、评分、推荐缓存和用户显式偏好。

系统内部调用关系是：

```text
Vue 前端 -> Spring Boot 后端 -> Python 推荐算法服务 -> MySQL
                         -> DeepSeek API
```

前端不会直接访问算法服务和 DeepSeek，所有业务都由后端统一编排，这样接口边界更清晰，也便于后续部署和维护。

## 3. 数据库设计

数据库名为 `movie_recommender`，核心表包括：

- `users`：保存用户信息。
- `movies`：保存 MovieLens 电影信息。
- `ratings`：保存用户评分，是推荐算法最主要的输入。
- `recommendations`：保存推荐缓存和推荐理由。
- `user_movie_preferences`：保存用户显式偏好，包括想看、收藏和不感兴趣。

其中 `ratings` 表支撑传统推荐算法，`recommendations.reason` 支撑 LLM 推荐理由持久化，`user_movie_preferences` 支撑用户反馈闭环。

## 4. 推荐算法链路

算法服务采用 Python FastAPI 实现，直接读取 MySQL 的 `ratings` 表。推荐流程是：

1. 从 MySQL 读取用户-电影评分数据。
2. 构造用户评分矩阵。
3. 根据相似用户偏好和 SVD 隐语义特征计算候选电影。
4. 对冷启动或数据不足情况使用热门高分电影兜底。
5. 返回 `movieId`、推荐分和算法侧推荐说明给后端。

需要说明的是，推荐结果中的 `score` 是算法排序分，不等同于用户评分满分 5 分。它用于比较推荐优先级，因此可能不是传统意义上的 0.5 到 5.0 用户评分。

## 5. 用户反馈闭环

为了让系统更像一个完整的推荐系统，我们不仅支持评分，还加入了显式偏好反馈：

- 用户可以给电影评分。
- 用户可以把电影标记为想看。
- 用户可以收藏电影。
- 用户可以标记不感兴趣。

其中“不感兴趣”会影响后续推荐展示。后端读取推荐缓存时，会过滤当前用户已经标记为不感兴趣的电影；前端也会立即把该电影从当前推荐列表中移除。

这样系统形成了一个闭环：

```text
用户行为 -> 数据入库 -> 推荐更新 -> 前端展示 -> 用户继续反馈
```

## 6. 前端功能展示

前端目前采用菜单式应用结构，主要包括：

- 推荐结果：展示个性化推荐、推荐分、结构化推荐依据、可信度标签和 DeepSeek 推荐总结。
- 电影库与评分：支持搜索、分页、跳页、首字母筛选、类型筛选、电影详情和评分提交。
- 我的评分：支持评分统计摘要、排序筛选、页内快速改分和删除评分。
- 我的画像：展示评分数量、平均评分、喜欢的类型、最近评分电影和显式偏好列表。
- 数据看板：展示电影规模、评分规模、类型分布、热门电影、用户评分画像和推荐画像。
- 智能问答：支持问题模板、上下文说明、DeepSeek 问答和本次问答历史。

## 7. LLM 接入设计

本项目接入 DeepSeek，采用 OpenAI-compatible API 格式。LLM 在系统中的定位不是直接参与推荐排序，而是作为语义增强层，主要负责：

- 解释为什么推荐某部电影。
- 根据用户画像回答观影问题。
- 把算法结果转化为更自然的语言。

为了避免大模型脱离系统数据自由发挥，后端会在调用 DeepSeek 前组装上下文，包括当前推荐排名、推荐分、推荐标签、用户高分和低分电影、想看、收藏、不感兴趣列表。因此当用户问“当前推荐第一名”时，LLM 能指向真实推荐列表的第一名；当用户问“根据我的评分推荐”时，LLM 能参考真实评分摘要，而不是凭空回答。

如果未配置 DeepSeek API Key，系统会自动进入离线模式，返回默认解释，保证推荐主链路仍然可运行。

这体现了我们在架构上的解耦原则：

```text
推荐计算归算法服务，语义解释归 LLM，业务编排归后端，展示交互归前端。
```

## 8. 数据可视化

为了增强答辩展示效果，我们加入了数据看板。它主要展示：

- 系统总览：电影总数、用户总数、评分总数、平均评分。
- 电影类型分布：说明电影库覆盖范围。
- 热门电影：展示 MovieLens 数据中交互最集中的电影。
- 我的评分画像：展示当前用户评分分布和类型偏好。
- 推荐画像：展示当前推荐列表的类型倾向和推荐分分布。

这些图表不是为了装饰，而是为了说明推荐系统的数据基础和推荐结果的可解释性。

## 9. 测试与运行

项目提供了本地一键脚本：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\deploy\scripts\start-local-fullstack.ps1
```

该脚本会启动 Python 算法服务和 Spring Boot 后端。前端通过：

```powershell
cd frontend
npm run dev
```

启动。

同时我们提供了全栈测试脚本：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\deploy\scripts\test-local-fullstack.ps1 -SkipLlmCheck
```

测试覆盖 MySQL 连接、算法服务、推荐接口、用户注册登录、电影分页、电影详情、偏好反馈、评分提交、评分删除、用户画像和推荐刷新等核心链路。

## 10. 项目亮点

本项目的亮点主要有五点：

第一，架构清晰。前端、后端、算法服务、数据库和 LLM 服务职责分离，便于维护和扩展。

第二，数据链路完整。MovieLens 数据导入 MySQL，算法服务直接读取数据库，推荐结果再由后端缓存并返回前端。

第三，推荐闭环完整。系统支持评分、想看、收藏、不感兴趣，并把这些行为沉淀为用户画像和推荐反馈。

第四，可解释性较强。推荐结果不仅有自然语言理由，还有结构化推荐依据和可信度标签。

第五，工程化程度较高。项目包含启动、停止、导入数据和全栈测试脚本，便于在不同电脑上复现运行。
